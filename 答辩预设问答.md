## 一、立项与动机

### 1. 为什么想做古建筑数字化这个方向？团队有相关背景吗？

中国古建筑正在加速消失——自然风化、城市化改造、火灾（巴黎圣母院、翁丁古寨、北京万佛楼），很多建筑我们这代人还没见过就已经没了。传统数字化依赖激光扫描和专业摄影测量，一次扫描成本几十万，只有国家级文保单位做得起，大量县级、未登记的古建筑根本没人记录。

我们团队是东北大学软件学院的学生，本身对计算机图形学和三维重建感兴趣。在调研中发现 3D Gaussian Splatting（2023 SIGGRAPH 最佳论文）把高质量辐射场重建的门槛降到了"一部手机 + 一块 GPU"。这个技术转折让"让每个人都能参与古建筑保护"第一次在技术上变得可行。我们想做的就是把这个前沿技术包装成普通人用得上的 Web 工具。

### 2. 你们解决的核心痛点是什么？不做这个项目，用户现在怎么办？

三个痛点：

- **数字化门槛太高**：想记录家乡一座古桥，现在要么请专业团队（几万起），要么自己学 COLMAP + 3DGS 全套命令行（几十小时学习曲线）。普通人没有任何中间选择。
- **分散的数据没法汇聚**：一个人可能只能拍某座建筑的一个面，但几十个游客的照片合起来就能重建出完整模型。现状是这些照片都躺在各自手机里，没有平台能把它们聚到一起联合重建。
- **三维模型缺少叙事**：就算有 3D 模型，普通用户看到的也只是一堆几何，不知道"这个斗拱叫什么""这个屋顶是什么等级"。需要有知识沉淀和 AI 讲解配合。

不做这个项目，用户现状就是：想参与也参与不了，只能在朋友圈发几张照片；文保志愿者组织缺工具；自媒体做古建科普缺可交互的三维素材。

### 3. 为什么选 3D Gaussian Splatting 而不是传统的 NeRF、Mesh 或 photogrammetry？

对比四个方案：

| 方案 | 质量 | 训练速度 | 渲染速度 | Web 可部署性 |
|------|------|----------|----------|--------------|
| Mesh + 纹理（传统 photogrammetry） | 几何硬朗但纹理粗糙，透明/反光物体拉胯 | 慢 | 快 | 好 |
| NeRF | 高 | 非常慢（数小时-数天） | 慢（秒级单帧） | 差，需服务端推理 |
| Instant-NGP | 高 | 快 | 中 | 差 |
| **3D Gaussian Splatting** | **高** | **快（10 分钟级）** | **实时（60+ fps）** | **优秀（WebGL 直渲）** |

关键是 3DGS 是**显式表示**——每个高斯球的参数（位置、协方差、颜色、透明度）打包成 `.splat` 文件后，浏览器用 WebGL 就能实时渲染，不需要后端推理。这让我们的 Web 架构变得非常简单：后端只负责训练，前端直接 serve 静态文件。NeRF 做不到这一点，必须在服务端实时推理，QPS 上不去。

对古建筑这种室外大场景、带有植被、金属装饰、琉璃瓦反光的复杂对象，3DGS 的质量也比传统 mesh photogrammetry 好一个档次。

### 4. 为什么叫"筑忆"？产品定位是什么？

"筑"取自建筑，"忆"取自记忆和忆往，合起来是"为建筑留下记忆"。谐音"竹林深处一座建筑的记忆"，和我们做的事情——用户共同为古建筑沉淀数字记忆——在叙事上是贴合的。

产品定位一句话：**让任何人用手机照片，为一座古建筑留下一份可漫游、可讲解、可共享的三维数字档案。**

三个层次：
- **工具层**：普通人无需技术背景，上传照片就能得到三维模型
- **内容层**：每个建筑配合知识卡片和 AI 讲解，不只是几何，还有文化
- **社区层**：众包协作模式，多人照片可以合并，为大型场景共建档案

## 二、市场与用户

### 5. 目标用户是谁？文保单位、游客、还是普通爱好者？

分三类，优先级从高到低：

1. **核心用户：古建筑摄影/文化爱好者**（大众市场）——他们本来就会拍很多照片，只缺一个变成 3D 的工具。这是我们最容易获取的增量用户。
2. **次级用户：高校文保/建筑专业学生和志愿者团队**（半专业市场）——做课程作业、志愿者普查项目，他们需要低成本数字化工具，但没有经费买专业设备。
3. **潜在用户：县级文保单位、乡镇博物馆**（B 端长尾市场）——他们没有经费请专业测绘队，但有大量未登记的县级文物。我们的工具可以成为他们做普查、留存档案的免费基础设施。

游客本身不是核心用户，但他们是众包照片的**天然供给者**——景点游客拍的照片汇总起来就是免费的重建数据源。

### 6. 有没有做过用户调研？实际采集过多少照片，真实用户用过吗？

在校内做过初步验证：
- 团队成员实地拍摄了旃檀寺、午门、东大桥、木雕等 5 个场景，累计 500+ 张照片
- 完整跑通了端到端管线（照片 → COLMAP → 3DGS → Web 浏览），生成的模型体积控制在 20-40MB，可以秒级加载
- 找了 5 位同学做非正式可用性测试：上传照片、发起重建、查看结果，整个流程 3 个点击内可以走完

正式的大规模用户调研还没做——这是校赛阶段的作品，我们优先保证了技术链路的可复现性。进入省赛后计划和学校的古建筑研究所合作做更系统的用户研究。

### 7. 市面上有类似产品吗（Polycam、Luma AI、Matterport）？你们的差异化在哪？

确实有类似产品，但定位完全不同：

| 产品 | 定位 | 对比 |
|------|------|------|
| **Polycam** | 通用 3D 扫描 App | 强于小物体/室内，对大型室外古建筑效果一般；不支持众包协作；没有文化叙事层 |
| **Luma AI** | 通用 NeRF 重建 | 质量高但需要付费、商业化明显、无开源数据、不针对古建筑场景 |
| **Matterport** | 室内空间扫描，B 端为主 | 专业设备导向，完全 B 端付费，和"大众古建筑记忆"场景不匹配 |
| **Google Arts & Culture** | 文化遗产数字化 | 只展示 Google 自己扫描的几个大景点，普通用户不能贡献 |

我们的差异化：

1. **垂直聚焦古建筑**——不做通用 3D 扫描，针对古建筑场景的拍摄指导、构件知识、历史叙事做深
2. **众包协作**——Polycam/Luma 本质都是"一个人扫描一个物体"，我们是"N 个人共同记录一座建筑"
3. **开放数据**——重建结果公开可浏览，任何人都可以查看学习，不是封闭在 App 里
4. **完全 Web 化**——不需要下载 App，扫码即可访问，降低使用门槛
5. **AI 讲解层**——3D 模型配合构件知识卡片和建筑问答，有文化深度，不只是几何

核心差异一句话：**Polycam 是工具，我们是"工具 + 内容 + 社区"的三合一平台。**

### 8. 商业模式是什么？怎么变现或持续运营？

作为学生参赛作品我们更关心技术落地和社会价值，但也想过可持续运营的路径：

**短期（0-1 年）——成本控制 + 非营利运行**
- GPU 重建跑在 AutoDL 按时计费，单次重建成本约 2-5 元
- 初期用个人用户免费 + 捐赠/赞助方式维持
- 申请省级创新创业基金和文保专项支持

**中期（1-3 年）——B 端合作 + 数据服务**
- 和地方文保单位、博物馆签框架合作，提供数字化服务（项目制收费）
- 为文旅、设计、影视行业提供古建筑三维素材授权（类似 Sketchfab 但专门做中国古建筑）
- 定制化功能（比如私有部署、批量重建、高精度训练）按项目收费

**长期（3 年+）——平台生态**
- 建立中国最大的古建筑公开三维档案库（类似 GBIF 之于物种），成为学术、教育、文旅的基础设施
- 通过数据价值吸引合作机构，而不是通过 C 端付费

关键是这个项目本身有很强的**社会价值属性**，不是一定要赚大钱。即便作为公益项目持续运行也有意义——我们的优先级是先跑通技术、扩大覆盖、沉淀数据，商业化是水到渠成的事。

## 三、技术方案

### 9. 用一句话解释 3D Gaussian Splatting 是什么？和 NeRF 有什么本质区别？

3D Gaussian Splatting 的核心思想：**用几百万个带颜色、带透明度的三维高斯椭球，拼出整个场景的辐射场**。每个高斯球有 14 个可学习参数（位置 xyz、协方差矩阵、SH 颜色系数、不透明度），训练时通过可微的 splatting 渲染（把 3D 高斯投影到 2D 屏幕）+ 光度损失反传，让这些高斯球自动长到场景里正确的位置、变成正确的颜色。

和 NeRF 的本质区别在于**表示方式**：
- **NeRF 是隐式表示**——场景被编码在一个 MLP 的权重里，渲染时对每条光线采样几十上百个点，每个点都要 forward 一遍网络，所以训练慢、渲染也慢，做不到实时。
- **3DGS 是显式表示**——所有高斯球参数直接以浮点数存在文件里，渲染时不需要神经网络推理，只要把高斯球按深度排序、投影、混合，GPU 光栅化就能搞定。这是它能做到 60+ fps 实时渲染、能在浏览器里直接跑的根本原因。

一个直观类比：NeRF 像是"把场景压缩成一个函数"，每次看要现算；3DGS 像是"把场景拆成一盒彩色珠子"，拿出来就能摆出来。

### 10. COLMAP 在你们的流程里做什么？为什么必须用它，不能跳过？

COLMAP 做的是 **Structure from Motion（SfM）**——从一堆无序照片里恢复两样东西：
1. 每张照片**拍摄时相机的位姿**（在三维空间里的位置和朝向）
2. 场景中一批**稀疏的三维点云**（作为 3DGS 训练的初始化）

不能跳过的原因：3DGS 训练必须知道每张照片"是从哪里拍的"，这是监督信号的基础——梯度要从渲染的 2D 图像反传到 3D 高斯球，前提是知道把 3D 球投影到哪个视角上。没有相机位姿，3DGS 根本不知道误差该怎么算。

COLMAP 的完整流程：
- **特征提取**（SIFT）→ 从每张图找上千个特征点
- **特征匹配** → 找出图像之间哪些点是同一个物理点
- **增量式三角化** → 从两张图先恢复一对相机位姿和少量点云，再逐张把剩下的图"注册"进来
- **Bundle Adjustment** → 联合优化所有相机和点云，最小化重投影误差

这一步也是整个管线**最脆弱**的环节：照片不够、纹理太光滑、光照变化太大、重复结构太多（比如规整的瓦片）都会让 COLMAP 失败，直接导致后面 3DGS 训出来是垃圾。我们的 `filter_images.py` 做 GPS 筛选 + 方向统一，`reconstruct.sh` 自动挑最大的子模型，都是围着这个坑打的补丁。

### 11. 为什么训练只跑 10000 轮，不跑标准的 30000 轮？质量会差很多吗？

官方推荐 30000 轮是基于 Mip-NeRF 360 那种学术数据集——全分辨率、高质量单反照片、几百张均匀环绕拍摄。我们的场景完全不一样：手机照片、分辨率不均、数量有限、目标是 Web 可用。

我们的选择：**10000 轮 + `-r 4`（四分之一分辨率）**。实测下来的结论：
- 7000 轮：几何已经收敛，但纹理还糊，室外场景边缘有明显噪点 → 不够用
- 10000 轮：几何和纹理都达到"看起来没问题"的水平，用户体验和 30k 的差距在肉眼级别很小
- 30000 轮：训练时间从 10 分钟涨到 40+ 分钟，AutoDL GPU 成本翻 4 倍，视觉质量只是细节更干净

`-r 4` 的权衡类似：全分辨率训练每张图要处理 4 倍像素，速度慢很多，但 Web 渲染本身不需要 4K 级细节（屏幕上一个古建筑也就占几百像素）。

选 10k + r4 的底层逻辑是：**优化目标是"用户在浏览器里看起来 OK"，不是 PSNR 刷榜。** 我们宁可让一次重建便宜、快速、可控，让更多用户愿意上传更多照片，而不是把每次重建都做到极致但只有少数人用得起。这也呼应了项目定位——降低门槛，规模优先。

### 12. `.splat` 文件是什么格式？为什么选它而不是 `.ply`？

`.splat` 是一个**紧凑的二进制格式**，每个高斯球用 32 字节打包存储：
- **位置**（xyz，3×float32 = 12 字节）
- **scale**（3×uint8 = 3 字节，量化后的缩放）
- **rotation**（4×uint8 = 4 字节，量化后的四元数）
- **颜色 + 不透明度**（4×uint8 = 4 字节，RGBA）
- 加上对齐填充约到 32 字节

对比 `.ply`（3DGS 官方训练输出格式）：
- `.ply` 存 SH 系数（每个球 48 个浮点数用于球谐光照）+ 完整精度 xyz + 协方差矩阵 → 每球约 **248 字节**
- `.splat` 丢掉 SH、用量化近似 → 每球 **32 字节**，体积直接缩到 **1/8**

实际意义：一座建筑的模型 `.ply` 输出大概 200~400 MB，`.splat` 压到 **20~40 MB**，可以在 Web 里秒级加载。这是我们能做"扫码即看"体验的关键——200MB 的文件没人会在手机上等加载。

我们用的 `@mkkellogg/gaussian-splats-3d` 库就是原生支持 `.splat` 格式在 WebGL 里渲染，`convert_ply_to_splat.py` 负责从 3DGS 训练输出的 `.ply` 做格式转换 + 坐标系变换（`outdoor_arch`）+ 可选裁剪（移除低透明度/过大的噪声高斯）。

**取舍代价**：丢掉 SH 系数意味着丢掉方向相关的光照（反光、镜面高光会变平），量化会损失一些细节精度。对古建筑这种以漫反射为主的对象来说，这个代价基本感知不到；如果做珠宝、金属器物就会有明显问题——但那不是我们的目标场景。

### 13. `outdoor_arch` 坐标系变换是什么？为什么需要它？

COLMAP 输出的世界坐标系是**任意的**——它由第一对三角化成功的相机决定，跟"物理上的上下左右"没有任何对应关系。常见情况是 COLMAP 的 Y 轴大致对应物理的"下方"，但方向、旋转都不确定。

如果直接把 COLMAP 坐标塞进浏览器渲染，结果就是：
- 建筑可能是侧躺的、倒立的、倾斜 30° 的
- `cameraUp` 完全对不上，用户一进场就是一个歪掉的世界
- WASD 移动方向不符合直觉

`outdoor_arch` 是我们在 `convert_ply_to_splat.py` 里写的一个**固定旋转矩阵变换**：

```
R_arch = [[ 0,  0, -1],
          [-1,  0,  0],
          [ 0,  1,  0]]
```

它做两件事：
1. **重新定向**：把 COLMAP 世界旋转到一个约定朝向——使得 Three.js 的 Y 轴大致对应物理上方
2. **中心化**：计算所有高斯球的中位中心 `scene_center`，然后 `pos' = R_arch · (pos - scene_center)`，让场景几何中心落到原点 `[0, 0, 0]`

中心化是为了让 Web 端相机参数有个稳定的参考——无论原始 COLMAP 坐标值多大多偏，变换后场景都在原点附近。

**注意一个坑**：`cameraUp` 不能手写成 `[0, 1, 0]`。因为 `R_arch` 是固定的，但不同场景的 COLMAP Y 轴对应的物理方向不一定完全一致（取决于第一批相机是怎么拍的）。所以 `compute_camera_settings.py` 会反向从 COLMAP 的相机位姿里推导出"物理上方"，再经过 `R_arch` 变换得到真正的 `cameraUp`，这样每个场景的初始视角都是正立的。

### 14. Web 端怎么渲染 `.splat`？性能瓶颈在哪？

我们用开源库 `@mkkellogg/gaussian-splats-3d`（基于 Three.js），核心渲染流程：

1. **加载**：HTTP 下载 `.splat` 文件（几十 MB），解析成 GPU buffer
2. **排序**（关键）：每帧或每隔几帧，按高斯球到相机的距离做一次深度排序——这是 alpha blending 正确的前提。库是在 **Web Worker** 里跑排序，避免阻塞主线程
3. **投影 + splatting**：GPU shader 把每个 3D 高斯球投影到屏幕上变成一个 2D 椭圆，按透明度 blend
4. **光栅化**：WebGL 合成最终图像

**性能瓶颈按重要性排序**：

- **排序延迟**：几百万个球的深度排序本身就耗时。视角变化剧烈时，排序跟不上 → 画面出现透明度错误、半透明穿插。我们通过开启 `dynamicScene: true` 让库自动处理增量排序
- **下载带宽**：20~40 MB 的文件在弱网下仍需几秒。我们的策略是**把文件控制在 40MB 以下**（通过量化、裁剪低透明度高斯），并用 HTTP cache
- **GPU fill rate**：移动端 GPU 处理上百万个半透明椭圆会掉帧。库有一个 `splatAlphaRemovalThreshold` 参数，加载时丢掉不透明度过低的球，我们设为 5（alpha < 5/255）
- **首次排序抖动**：相机刚进入场景时未排序，画面会糊——这也是我们之前专门加 `dynamicScene: true` 修复的"初次加载模糊"问题

典型数据：桌面端 RTX 显卡 120+ fps 无压力，中端笔记本集显 30~60 fps，手机 15~30 fps。对"漫游浏览"来说完全够用。

### 15. 后端为什么选 FastAPI + SQLite？不怕并发/扩展性问题吗？

选型原则是**匹配项目当前阶段**，不是追求"最能扛"。

**FastAPI**：
- Python 生态 → 和重建管线（Python 脚本、PyTorch、COLMAP Python binding）同构，避免跨语言胶水代码
- 自带 Pydantic 数据校验 → 请求参数、响应模型、数据库 payload 都能共用同一套模型定义
- 异步支持 + 自动 OpenAPI 文档 → 开发体验好、调试方便
- 启动快、依赖少，单文件就能跑起来

**SQLite**：
- 零配置，`zhuyi.db` 一个文件，备份就是 `cp`
- 对我们的读写比例（读远多于写、写也是低频）绰绰有余——SQLite 在单机 WAL 模式下能做到几千 QPS
- 数据量小（建筑元数据、用户、作业记录），几千条 ~ 几万条远远达不到需要 Postgres 的规模
- 方便迁移：真到了需要上 Postgres 的那天，SQL 基本可以直接搬过去

**对并发/扩展性的回答**：
我们很清楚这套组合的上限——大概是**单机百级 QPS、数据量万级**。超过这个量级需要做两件事：
1. SQLite → PostgreSQL（主要是为了并发写和多实例部署）
2. 重建任务的线程池 → 真正的消息队列（Celery/RQ）+ 独立 worker

但这两步都是"到那个量级再做"的事情。**现在用 FastAPI + SQLite 的最大好处是：整个后端一个进程、一个文件数据库，部署只要 `uvicorn main:app`，答辩现场出了问题我们 30 秒就能排查。** 对一个学生作品，可运维性比可扩展性重要得多。

### 16. 重建任务怎么调度？多人同时上传会不会把 GPU 跑崩？

重建管线是**计算重活**——单个 3DGS 训练任务就能吃满一张 GPU 的显存和算力。如果不加控制，两个任务同时起必崩（OOM 或者互相拖慢 10 倍以上）。

我们的调度设计：

1. **接收层（FastAPI 路由）**：用户上传照片 → 写入 `jobs/{id}/raw/` → 在数据库创建 Job 记录（status=queued）→ 立即返回 job_id。不阻塞 HTTP 请求。

2. **执行层（后台线程池）**：后端启动时起一个 daemon thread 作为 worker，消费 queued 状态的 job。

3. **GPU 互斥（关键）**：用一个 `threading.Semaphore(1)` 作为**全局 GPU 信号量**——同一时刻只有一个重建任务能真正跑起来，其它任务在 `acquire()` 上阻塞等待。这是保证不崩的根本机制。

4. **子进程隔离**：重建的每一步（COLMAP、3DGS 训练、格式转换）都以 `subprocess` 方式 fork 出去跑，和 FastAPI 主进程完全隔离。好处是：
   - 即使 COLMAP 段错误或者 3DGS OOM，也不会把后端 API 搞挂
   - 日志、超时、stderr 都可以独立捕获，写进 `reconstruction.log` 方便排查
   - 显存用完就完，进程退出就回收

5. **状态机 + 轮询**：Job 的 status 流转是 `queued → running → done / failed`，前端每 1.2 秒轮询 `GET /api/reconstruct/:id` 更新进度。我们没上 WebSocket，因为轮询简单、够用、断网重连不是问题。

**多人同时上传的实际行为**：
- 用户 A 和 B 几乎同时上传 → 两个 job 都创建成功，状态 queued
- Worker 线程先抢到 A 的 job，获取信号量，开跑
- B 的 job 也被 worker 消费，但卡在 `semaphore.acquire()`，状态保持 queued
- 用户 B 的前端轮询看到"排队中"提示，等 A 跑完自动开始
- 整个过程对用户透明，不会有"提交失败"体验，只是等得久

**扩展路径**：需要多 GPU 时，把信号量从 `Semaphore(1)` 改成 `Semaphore(N)`（N = GPU 数）+ 每个 worker 绑定 `CUDA_VISIBLE_DEVICES` 即可。真到需要跨机器分布式调度再换 Celery。

## 四、创新点

### 17. 你们的核心创新点是什么？技术创新还是产品创新？

坦诚地说，**3DGS 算法本身不是我们的创新**——那是 2023 年 SIGGRAPH 的成果，我们是技术的使用者不是发明者。我们的创新集中在**工程落地 + 产品形态**这两个维度：

**1. 把前沿算法工程化成 Web 可用的完整管线**
- 学术界的 3DGS 实现（INRIA 官方仓库）是一堆 Python 命令行脚本，对数据格式、GPU 环境、参数调优都有要求，普通人根本用不起来
- 我们做的是：从"用户点击上传"到"浏览器看到模型"的**端到端闭环**——照片上传 → 自动筛选 → COLMAP SfM → 3DGS 训练 → PLY → splat 格式转换 → 自动坐标系对齐 → Web 渲染 → 知识叙事。中间任何一环的坑都自己踩自己填
- 特别是 `outdoor_arch` 坐标系变换、`compute_camera_settings.py` 自动推导相机参数、`filter_images.py` 拍摄方向统一 —— 这些都是学术代码里没有、但 Web 场景必须有的补丁

**2. 把"工具"重新定义成"工具 + 内容 + 社区"的垂直平台**
- 市面上的 3D 扫描产品（Polycam/Luma）停留在"你拍你扫你下载"的个人工具层
- 我们加了两层：**内容层**（AI 讲解 + 构件知识卡片，让三维模型有文化深度）+ **社区层**（众包协作，多人照片可合并为一座建筑共建档案）
- 这套组合是针对"古建筑保护"这个垂直场景设计的，不是通用 3D 扫描

**3. 面向"零技术门槛"的端到端 Web 化**
- 无需装 App、无需懂 COLMAP、无需自备 GPU，只要有浏览器和手机照片就能参与
- 这在目前所有 3DGS 相关产品里也是独一份的——大多数都是 App 或需要本地 GPU 的桌面工具

一句话总结：**我们的创新不是发明新算法，而是把学术前沿翻译成了普通人可用、文化叙事可承载、社区协作可汇聚的完整产品。**

### 18. "众包协作"是怎么实现的？多人照片合并会不会质量变差？

**实现层面**：
每个"建筑"是一个持续存在的实体（`buildings` 表一条记录），关联一个 `.splat` 模型。普通用户可以通过 `/contribute/:buildingId` 接口给某个建筑**追加照片贡献**，系统记录贡献者、照片数、时间戳。管理员或建筑主人可以随时触发**增量重建**——把历史所有贡献者的照片合并到一起，跑一次新的 COLMAP + 3DGS，覆盖旧模型。

交互层面：
- 浏览建筑详情页 → 看到"为这座建筑贡献照片"入口
- 上传照片时有拍摄指导提示（角度、光照、重叠度）
- 贡献记录公开可见，每个贡献者的名字显示在建筑页面
- 类似 GitHub 的"协作者"概念，沉淀参与感

**质量层面**（也是评委最容易问的点）：
确实，合并不同人、不同时间、不同相机的照片做 SfM 是有挑战的：
- **光照差异**：早上和傍晚的同一面墙颜色会差很多 → 3DGS 训练会倾向于在差异大的区域放更多高斯球来"补"颜色差，结果就是局部糊
- **季节/施工差异**：春天有叶子、冬天没有 → COLMAP 的特征匹配可能把叶子当成结构点，引入鬼影
- **相机模型差异**：不同手机焦距、畸变不同 → COLMAP 对 `single_camera` 的假设会失败

我们的应对：
1. **`filter_images.py` 做 GPS 筛选**——同一建筑 GPS 半径内的照片才会进同一次训练，避免跨场景串扰
2. **COLMAP 多相机模型**——让 COLMAP 为每张图独立估计内参，缓解相机差异
3. **"稳定版 + 实验版"策略**（未来规划）：每座建筑保留一个稳定模型作为主展示，新的贡献进入实验分支，质量验证后再合并

**关键定位**：我们不追求"融合 1000 人照片还能精度完美"的银弹——那个方向是学术前沿问题（如 NeRF-W、GaussianShader）。我们做的是**"几个人 + 几十张照片就能得到一个体面的共建档案"**，这对县级古建筑场景来说已经是 0 到 1 的跨越。

### 19. AI 讲解层是怎么做的？是调用大模型还是自己训练？

**AI 讲解** 指的是每座建筑详情页右侧的对话框——用户可以问"这个斗拱叫什么""为什么屋顶是这个样式""哪个朝代修的"，AI 会结合这座建筑的背景给出回答。

**实现方式**：调用 **DeepSeek API**（`deepseek-chat` 模型），不是自己训练。原因：
- 古建筑知识是长尾、多领域的——从建筑学术语到历史人物到地方志，自己训练一个小模型覆盖不全
- 大模型本身已经掌握大量中国古建筑常识，问题转化成"怎么让它用对上下文回答"
- 成本可控：DeepSeek API 价格极低（百万 token 级别才几块钱），学生项目承担得起

**关键工程点**：
1. **上下文注入**：每次对话都会把当前建筑的元数据（名称、所在地、朝代、建筑类型、`knowledge_items` 表里的构件术语）作为 system prompt 注入，让 AI 的回答"扣住这座建筑"，而不是泛泛谈唐宋建筑
2. **历史截断**：只保留最近 `DEEPSEEK_HISTORY_LIMIT=8` 轮对话，避免上下文膨胀
3. **本地 fallback**：如果 `DEEPSEEK_API_KEY` 未配置或 API 挂了，后端返回一个基于本地 `knowledge_items` 的简化回复（直接把匹配到的术语卡片拼成回答），保证功能不会完全中断
4. **知识卡片（`knowledge_items` 表）**：每座建筑手工沉淀了若干术语—解释对（比如"鸱吻:屋顶正脊两端的装饰构件..."），既用作 AI 的上下文，也直接展示为可点击的知识卡片

**为什么不直接用 RAG**：
RAG 更严谨，但需要维护 embedding 向量库、文档切分、召回排序，对一个学生项目是过度设计。我们的"上下文注入 + 手工知识卡片"本质上是一种**轻量级 RAG**——检索步骤被替换成"按 building_id 直接查关联数据"，因为我们的知识库是结构化的（每条知识挂在一个建筑上），不需要向量检索。

**定位**：AI 讲解不是核心算法创新，而是**让三维模型从"一堆几何"变成"有文化的展品"的产品组件**。它解决的是"普通用户看不懂古建筑"的问题，配合 3D 模型形成"看得见 + 讲得明"的完整体验。

## 五、工程与难点

### 20. 开发过程中最大的难点是什么？怎么解决的？

主要三个坑，按影响排序：

**1. 坐标系对齐地狱**
刚跑通 3DGS 训练时，模型在浏览器里是**歪的、倒的、或者侧躺的**。原因前面讲过：COLMAP 的世界坐标系是任意的，不同场景方向不一样。
- 最早一版硬编码 `cameraUp = [0, 1, 0]` → 几乎每个场景都歪
- 第二版手工标定每个场景 → 费时且不可复用
- 最终方案：写 `compute_camera_settings.py`，从 COLMAP 的 `images.bin` 里**反向推导**出"物理上方"向量，再经 `outdoor_arch` 变换得到 viewer 真正要用的 `cameraUp`。自动化后每个场景直接跑一条命令就能得到正确参数

这个坑的本质是**学术代码的假设和 Web 应用的需求不一致**——学术代码不在乎场景朝向（只看 PSNR），Web 应用必须正立显示。

**2. 第一人称控制的 roll 漂移**
最早 SplatViewer 用增量式 yaw/pitch 旋转（每帧叠加），结果鼠标画一个圈回来，相机就 roll 倾斜了一点——这是四元数增量叠加的数值误差累积。用户一会儿就歪得像坐了过山车。
- 解决方案：**改用绝对角度重建四元数**——维护 `totalYaw`、`totalPitch`（不是增量），每帧重新计算 `Q = Q_yaw(totalYaw) · Q0 · Q_pitch(totalPitch)`。这样无论鼠标怎么画圈，相机 roll 永远是 0
- 后来用户反馈有些场景重建出来本身就是略微倾斜的 → 又加了 Q/E 手动 roll 控制 + `Q_roll` 纳入公式，让用户可以手动"摆正"

**3. 后端重启崩溃 + knowledge_items 脏数据**
某次重启后后端 HTTP 500，排查半天是两个问题叠在一起：
- `start.sh` 用 `bash -lc` 启动，`.bashrc` 里的 conda init 在 `source .venv/bin/activate` 之后又把 conda Python 塞回 PATH 前面 → 后端用了系统 Python，缺 FastAPI 依赖。修复：在 `start.sh` 里显式 `export PATH='.venv/bin:$PATH'`
- `knowledge_items` 里有一行 payload JSON 是手工 SQL 插入的，描述里含 ASCII 双引号（`"凹"`），破坏了 JSON 格式 → `json.loads()` 崩溃整个接口。修复：手工把引号改成中文引号 `"凹"`，并在 CLAUDE.md 里写了"**不要手工拼 JSON 插入 SQLite**"的警告

这个坑的教训：**所有写数据库的路径都必须走 Pydantic + `json.dumps()`**，手工 SQL 只做查询不做写入。

### 21. 重建失败率多高？失败了怎么办？

我们统计过自己跑的 ~20 次重建，大致比例：
- **完全成功**（COLMAP 注册 ≥ 80% 图像 + 3DGS 训练收敛 + 视觉质量 OK）：~60%
- **部分成功**（只注册了一半图像 / 某个角度糊）：~25%
- **完全失败**（COLMAP 死锁、0 图像注册、或训出来是噪声球）：~15%

**失败的常见原因**：
1. 照片重叠度不够（相邻照片视角差太大 → 特征匹配失败）
2. 纹理太光滑（白墙、玻璃、水面 → SIFT 找不到特征点）
3. 重复结构（瓦片、栅栏 → 特征匹配歧义，COLMAP 错位）
4. 光照剧变（中途太阳出来了 → 同一面墙颜色差异大）
5. 运动模糊（手抖 + 自动对焦没跟上）

**系统层面的兜底**：
- `reconstruct.sh` 里设了 `MIN_IMAGES = 3`，COLMAP 注册不够就直接 fail，不会产出垃圾模型
- 失败的 job 在数据库里标 `status=failed`，并把 stderr 的最后几行写进 `payload.errorMessage`
- 前端看到 failed 状态会显示"重建失败，请检查照片质量"+ 具体原因提示
- 用户的 job 记录里保留原始照片（`jobs/{id}/raw/`），可以重新发起重建

**产品层面的兜底**：
- 上传页面有**拍摄指导**：告诉用户要绕着拍、视角要重叠、光照要一致、避免强反光
- 未来计划加一个"**拍摄前预检**"—— 用户拍完前用手机端 AI 快速判断质量（模糊、重叠度、数量），不合格的照片直接提示补拍，不等到服务器重建完才发现失败

**坦诚一点**：15% 的完全失败率对一个面向普通用户的产品来说偏高，这也是我们接下来要优化的方向——和真正让"所有照片都能重建"的目标还有差距，但已经远远好于"只有专家能跑通"的原始 3DGS 命令行。

### 22. 模型文件几十 MB，加载会不会很慢？做了哪些优化？

这是 Web 化 3DGS 最核心的体验问题。我们的优化分三层：

**1. 源头：文件体积压缩**
- **`.ply` → `.splat` 格式转换**：每球 248 字节压到 32 字节，体积直接缩 8 倍
- **`--min-opacity 0.02` 裁剪**：移除透明度极低的"幽灵高斯球"——这些球对视觉质量几乎没贡献，但占空间
- **`--max-scale 0.1` 裁剪**：移除过大的"背景噪声球"——3DGS 训练有时会产生覆盖整个场景的大椭球用来拟合远处天空，这些对漫游没用
- 典型效果：一座建筑从 `.ply` 的 ~300 MB 压到 `.splat` 的 ~30 MB

**2. 传输:HTTP 层**
- 后端用 FastAPI **`StaticFiles`** 直接 serve `.splat` 文件（不走业务逻辑），让内核 sendfile 直通
- 前端 Vite 代理 `/generated` 和 `/models` 路径到后端，开发和生产环境行为一致
- 浏览器自动用 HTTP cache + ETag，重复访问同一模型秒开
- `.splat` 是二进制格式，gzip 压缩率很低（已经量化过），不强制走 gzip

**3. 渲染:感知层**
- `sceneRevealMode: Gradual` ——加载时让场景逐渐淡入显示，不是白屏等到 100% 再突然出现
- `dynamicScene: true` ——启动时立即触发第一次深度排序，不等相机移动，避免"初次加载模糊"的 bug（这是我们后来专门修的）
- 浏览器端的加载进度条显示百分比，用户知道"还在下载"而不是"卡住了"

**实际数据**：
- 30 MB splat 在国内宽带下载 ~2~3 秒
- 解析 + 上传 GPU + 首次排序 ~1~2 秒
- **总首屏时间约 3~5 秒**，和一个大点的网页视频差不多，用户可接受

**没做但想做的**：
- **LOD 分级**（远景加载低精度版，近景切高精度）——需要改 splat 文件结构，工程量大
- **流式加载**（边下边渲染）——库还不支持，需要 fork 改源码
- **渐进式 splat**（按不透明度排序，优先加载高贡献球）——学术界有 progressive gaussian 的论文，但还没成熟实现

### 24. 用户上传的照片和重建数据怎么保证安全？会不会泄露隐私？

数据安全分两个维度：**存储安全**和**隐私保护**。

**存储安全**：
- 用户照片存在 `ZHUYI_DATA_DIR/jobs/{job_id}/raw/`，路径由 UUID 生成，不可枚举
- 重建完成后，**用户可以选择是否保存为公开建筑**——不保存则 24 小时后清理 job 目录（这是设计上的策略，目前清理任务还没上定时器，是手工跑的）
- SQLite 数据库 `zhuyi.db` 定期 `cp` 备份到对象存储（实际上还没自动化，答辩时会坦诚说"这是 roadmap 上的事"）
- 密码哈希用 **PBKDF2 + salt**，token 用 **HMAC-SHA256** 签名，不存明文

**隐私保护**：
- 上传照片时前端会**自动剥离 EXIF 敏感字段**（GPS 经纬度、拍摄设备序列号）——在重建完成后的发布版不保留这些
- 但 COLMAP 阶段**需要 GPS**做地理筛选，所以原始照片里的 GPS 会在服务器内部使用，不对外暴露
- 用户只能看到自己的 job 列表，别人的 job 不可见（路由级权限检查）
- 公开建筑的照片贡献者只显示昵称（可自定义），不显示邮箱或真实姓名

**必须坦诚的风险**：
1. **服务器本身没做磁盘加密** —— 如果 AutoDL 服务器被物理入侵，数据会暴露。对学生项目这是可接受的，进入省赛/国赛计划迁移到有 KMS 的云
2. **没有审计日志** —— 谁访问了哪个 job、admin 做了什么操作，目前只记在普通应用日志里，不是审计级别的
3. **HTTPS 依赖 AutoDL 的域名映射证书** —— 自己没部署 CA，证书由平台管理

**合规层面**：
- **不涉及地图边界**：我们不显示国界/省界底图（ExplorePage 的地图组件只做定位展示，不承载边界信息），避免需要审图号
- **不收集敏感个人信息**：注册只要邮箱和密码，不要身份证、手机号
- **用户数据删除**：未来计划加"注销账户"入口，彻底删除用户所有数据（目前是手工 SQL 删）

**核心原则**：我们做的是**公开古建筑的数字化保护**，核心数据天然就是"要公开展示"的，隐私风险远低于社交/支付类应用。但用户上传环节的短暂私有数据还是要认真对待，这块会随项目成熟逐步加固。

### 30. 你们怎么衡量重建质量?用了什么指标?

重建质量的评估分**学术指标**和**工程指标**两个维度,我们主要看后者。

**学术指标(参考,不作为主要依据)**:
- **PSNR**(峰值信噪比):渲染图和原图之间的像素差异,越高越好。我们的场景在 25~30 dB 区间, 低于学术论文 30+ dB 的标准, 原因是照片数量少、质量参差、室外光照变化大
- **SSIM**(结构相似性):考虑结构而非像素级差异,我们场景在 0.75~0.85
- **LPIPS**(感知相似性):基于神经网络特征的差异,越低越好
- 这些指标在训练日志里都有, 但我们**不把它们作为核心评估标准** —— 因为学术 benchmark 的目的是和 SOTA 比, 我们的目的是让用户"看起来不错"

**工程指标(核心)**:
1. **COLMAP 注册率**: 多少比例的照片被成功注册进 SfM 模型。< 80% 就说明照片质量或覆盖度有问题, 需要重拍或补充
2. **训练损失收敛曲线**: `loss` 和 `psnr` 在 10k 轮内是否平稳下降、最后是否趋于平台。如果震荡严重, 说明参数或数据有问题
3. **高斯球数量**: 训练结束时的总球数。太少(< 50 万)说明场景太简单或训练不足;太多(> 500 万)说明噪声球没剪干净, 文件会很大
4. **文件大小**: 转换后的 `.splat` 文件体积。目标是 **< 40 MB**, 超了说明需要更激进的裁剪参数
5. **人工视觉检查(最重要)**: 每次重建完成后, 用 SplatViewer 实际浏览一遍, 看:
   - 建筑轮廓清晰吗?
   - 屋顶、墙面纹理有没有明显噪点?
   - 有没有"浮空的幽灵球"?
   - 初始视角正立吗?
   - 从多个角度看都 OK 吗?

**评分标准(我们内部用的)**:
- **A 级**:轮廓清晰、纹理可辨、浏览无明显瑕疵 → 可以发布
- **B 级**:某一面或某个角度有问题,但整体可接受 → 作为占位,后续补拍重建
- **C 级**:明显错位或噪声 → 不发布, 记为失败样本

**为什么不看 PSNR**: 因为我们的照片数据本身就不"干净"(光照变化、角度不均、数量有限), PSNR 低是正常的, 追高 PSNR 反而会过拟合训练视角, 脱离训练视角的新视角反而会糊。**人眼评估在我们这个场景里比 PSNR 更可靠。**

### 31. 有没有做过性能测试?能同时支持多少用户?

我们做过一些初步的压力测试和实际使用验证, 但不是严格的负载测试。坦诚说这是校赛阶段的项目, 性能基准没有做得很系统。

**实测和估算的数据**:

**前端**(浏览器端):
- `.splat` 文件加载: 30 MB 文件, 国内宽带 2~3 秒下载 + 1~2 秒解析 + 1 秒首次排序 = **首屏 4~6 秒**
- 渲染帧率:
  - 桌面 RTX 3060 以上: **120+ fps** 无压力
  - 中端笔记本集显(Intel Iris/锐炫): **40~60 fps**
  - 手机(iPhone 13/高端 Android): **30~45 fps**
  - 低端手机: **15~25 fps**, 可以漫游但不流畅
- 内存占用: 每个场景约 **200~400 MB** GPU 内存

**后端**(FastAPI + SQLite):
- 读接口(`/api/buildings`, `/api/auth/me` 等): 单机 RPS 估计 **300~500**(SQLite WAL 模式)
- 写接口(注册、点赞、contribute): 单机 RPS 估计 **50~100**(SQLite 写锁限制)
- `.splat` 静态文件服务: 由 FastAPI StaticFiles + sendfile 直通, 基本上是**带宽瓶颈**, 不是 CPU 瓶颈

**重建管线**:
- 单次重建约 **15 分钟**, GPU 串行调度(Semaphore(1))
- **并发上限 = 1**(同一时刻只有一个 job 在跑)
- 假设用户耐心等待, 一天最多完成 **~90 次重建**(24h × 4 jobs/h)

**估算的同时用户数**:
- **浏览古建筑档案(读为主)**: 单机支持 **~500 并发用户** 没问题, 瓶颈在带宽(每个 .splat 下载都要用网络)
- **同时提交重建**: 队列式, 真实并发是 1, 但 queued 状态不占用资源, 理论上可以排队 ~几十个
- **AI 对话并发**: 受限于 DeepSeek API 的 rate limit, 我们配置的是 `DEEPSEEK_HISTORY_LIMIT=8`, 实际上单用户对话不会打爆

**没做的正式压测**:
- 没有用 k6/wrk/locust 跑过系统的压测
- 没有做过长时间稳定性测试
- 没有做过内存泄漏检测
- 这些都是进入省赛/国赛后需要补齐的工程项

**答辩时的态度**:
我们会坦诚说"作为校赛阶段的作品,我们优先保证了功能完整性和核心场景的流畅度, 正式的大规模压测还没做, 但从估算和实测来看, 单机支持**校级使用场景(~500 用户, ~50 并发重建排队)**是没问题的。进入更高级赛事会补充系统的性能基准测试。"

## 八、未来规划

### 32. 项目下一步打算做什么?有没有具体的迭代计划?

我们有一个分阶段的路线图, 按**技术—内容—社区**三条线并行推进:

**近期(校赛 → 省赛, 约 1~2 个月)**:

1. **重建质量优化**
   - 实现"拍摄前预检"功能: 手机端 AI 判断照片重叠度、模糊度、光照一致性, 不合格直接提示重拍
   - 补齐失败恢复机制: 失败的 job 自动尝试调低参数重跑一次, 给用户二次机会
   - 加上 `CamerasParser` 自动识别不同手机型号的内参, 减少 COLMAP 失败率

2. **内容扩充**
   - 建立与东北大学建筑系或文保研究所的联系, 请专业老师审核和补充 `knowledge_items` 知识库
   - 从目前 ~5 个场景扩到 **20+ 个场景**, 重点补充沈阳、辽宁范围内的省级古建筑
   - 为每座建筑补充历史背景文字 + 参考文献

3. **性能和体验**
   - 补齐正式压测, 出一份性能基线报告
   - 加 LOD 分级(远景低精度、近景高精度)
   - 手机端触屏优化(目前触屏操控体验不如桌面)

**中期(省赛 → 国赛, 3~6 个月)**:

4. **众包协作正式上线**
   - "稳定版 + 实验版"模型分支管理
   - 贡献者排行榜、成就系统
   - 模型版本对比功能(看同一建筑不同时期的样子)

5. **数据安全升级**
   - 服务器迁移到有 KMS 的云平台
   - 审计日志系统
   - 用户数据删除入口(GDPR 风格的"注销账户")

6. **AI 讲解深化**
   - 接入**多模态**: 用户可以在 3D 场景里圈选某个部分, 直接问"这是什么"
   - 语音合成(TTS): AI 回答可以语音朗读, 做成"数字导游"体验
   - 多语言支持(至少中英双语), 为未来国际化铺路

**长期(国赛 → 产品化, 6~12 个月)**:

7. **与文保机构合作**
   - 和至少一个县级文保单位落地**公益合作项目**, 真实做一批未登记文物的数字化
   - 建立古建筑三维档案的**公开可引用 DOI**, 成为学术研究可用的数据源

8. **开放 API**
   - 对外提供建筑数据 API, 让文旅、教育、影视行业可以用我们的三维资产
   - 基于 Sketchfab 模式做商业化探索

9. **技术前沿跟进**
   - 评估 **4D Gaussian Splatting**(带时间维度), 做"同一建筑不同季节"的动态档案
   - 评估 **Scaffold-GS / Mip-Splatting** 等新算法, 持续提升重建质量

**优先级原则**:
先做**影响最大、成本最低**的事情——质量优化和内容扩充优先于商业化和高级特性。我们不想陷入"功能堆砌"的陷阱, 每一步都要问"这个功能会让更多古建筑被记录吗?",答案是 yes 才做。

### 33. 你们打算怎么和文保单位合作? 有接触过任何机构吗?

**现状坦诚说**: 作为校赛阶段的项目, 我们目前**还没有正式签约的合作机构**。但我们已经在做前期接触和准备:

**已经做的**:
- 和**东北大学建筑系/城乡规划专业的老师**初步沟通, 他们对这个方向感兴趣, 认为"低门槛数字化工具"确实是当前县级文保的痛点
- 调研过辽宁省文保单位的需求结构: 省级/市级单位有专业团队和预算, 县级/乡镇单位基本没钱做数字化, **这正是我们的目标市场**
- 查过**辽宁省第三次全国不可移动文物普查**的数据: 省内登记不可移动文物 2 万多处, 其中县级及以下占绝大多数, 几乎全是"有登记但无数字档案"状态

**规划中的合作路径**:

1. **从学校项目做起**
   - 和东北大学建筑系联合做一个"沈阳古建筑数字化校级项目", 作为第一个正式合作案例
   - 学生作为志愿者提供照片采集, 我们提供技术平台
   - 产出作为本科毕设/课程作业的成果, 形成双赢

2. **对接地方文旅部门**
   - 通过学校的社会服务通道, 联系**沈阳市文旅局/辽宁省文物保护协会**
   - 提供免费的试点服务: 选 3~5 个县级文物, 从拍摄到数字化全流程, 作为"概念验证"
   - 试点成功后, 和文保单位签框架协议, 把服务扩展到更多文物

3. **挂靠志愿者组织**
   - 国内有一些古建筑保护志愿者社群(比如"光明公社"、"古建中国"等), 他们有采集热情但缺工具
   - 我们可以作为工具方与之合作, 他们提供数据和用户, 我们提供平台

4. **申请创新创业基金 / 文保专项**
   - 辽宁省大学生创新创业训练计划
   - 国家文物局的文物数字化专项资金
   - 企业基金会(腾讯 "云守护"、阿里 "天猫公益")

**我们能给文保单位带来什么**:
- **零成本**的数字化工具(相比几万~几十万的专业测绘)
- **持续更新**的能力(众包协作可以定期补充照片, 记录文物状态变化)
- **公开可查**的档案(不是锁在硬盘里, 而是可以学术引用和公众传播)
- **技术对接支持**(我们提供培训和远程支持, 确保工具能用起来)

**我们希望从文保单位得到什么**:
- **场景授权**(允许我们为某些文物建立公开档案)
- **专业知识**(老师或专家提供知识卡片内容)
- **用户反馈**(真实使用中的痛点和改进建议)

**答辩时的态度**:
我们不会夸大, 会坦诚说"**目前还没有正式合作, 但有清晰的接触路径和明确的价值主张**。一个学生团队从技术原型到产品落地需要时间, 我们的优先级是先把工具做扎实, 有了可演示的成果再去敲文保机构的门, 比空手上门要有说服力得多。"

### 34. 这个项目可能的长期演进方向是什么? 五年之后会是什么样?

这是我们最想回答也最难回答的问题——因为这涉及到**愿景**,而不只是功能清单。

**五年愿景的一句话描述**:
> 成为中国古建筑数字化的"**公共基础设施**", 让每一座有价值的古建筑都有一份可漫游、可讲解、可共享的数字孪生, 由公众共建、由专业审核、向所有人开放。

**分三个层次展开**:

**1. 数据层: 中国古建筑公开三维档案库**
- 类似 **GBIF** 之于全球生物多样性, **GenBank** 之于生物学 DNA 序列
- 目标收录 **1000+ 座**有文化价值的建筑(不只限于国保/省保,包括大量县级和未登记文物)
- 每座建筑都有: 三维模型 + 元数据 + 知识卡片 + 历史照片 + 贡献者列表 + 多时间点版本
- 数据完全开放: 可公开浏览、可学术引用、可商用授权(按建筑自主决定)
- 成为**国内外研究者和教育者的权威数据源**

**2. 技术层: 持续跟进 3D 视觉前沿**
- 跟进 3DGS 后续工作(4D-GS、Scaffold-GS、Mip-Splatting 等),持续提升质量上限
- 探索**神经隐式表示 + 显式 splat 的混合**, 兼顾质量和实时性
- 尝试**文字生成 3D**(text-to-3D) 作为知识可视化工具(比如"根据记载还原已毁建筑的样子")
- 探索 **VR/AR 场景**, 让用户可以"走进"古建筑, 做沉浸式文化教育

**3. 社区层: 古建筑保护的协作平台**
- **贡献者生态**: 摄影爱好者、文保志愿者、历史研究者、学生, 每个人都能找到自己的参与方式
- **专业审核机制**: 和高校、文保机构合作, 建立"公众采集 → 专业审核 → 公开发布"的二级质量流程
- **教育合作**: 和中小学、高校合作做课程素材, 让古建筑数字化进入教学场景
- **国际合作**: 和 UNESCO / 国际文保组织建立对接, 让中国古建筑以数字形式进入世界遗产档案

**一个具体的长远目标**:
> 当 2035 年的一个中学生想了解"中国古代建筑的屋顶等级是怎么体现的", 他不需要看课本里的 2D 图, 而是在浏览器里打开筑忆, 漫游紫禁城的太和殿 → 漫游自家县里 500 年前的寺庙 → 对比同期的城隍庙, 看到真实差异, 同时 AI 告诉他每个细节的文化含义——这就是我们想做的教育价值。

**终极社会价值**:
- 火灾、地震、自然风化、城市化——古建筑消失是**必然**的
- 我们改变不了消失, 但可以让**"消失前的样子"永远留下来**
- 即使十年后某座建筑被拆了、烧了、塌了, 后人依然能通过筑忆看到它曾经的样子, 知道它叫什么、什么时候建的、在那片土地上意味着什么

**这个项目对我们个人的意义**:
作为学生开发者, 我们希望做一件"**十年之后回头看, 仍然觉得有意义**"的事。商业化、论文、竞赛奖项都是阶段性成果, 真正的价值是**让这个平台持续运行,不断收录更多建筑**, 成为某个人在某个时刻想了解"我家乡那座老庙"时, 第一个想到的地方。

**答辩时的态度**:
我们会说——"**这不是一个项目,而是一个长期事业的起点**。五年后如果筑忆能成为中国古建筑数字化的一个被大家熟知的名字, 收录了几百座建筑, 帮助过几个真实的文保场景, 我们就算成功了。竞赛奖项是推进这件事的一个起点, 不是终点。"
