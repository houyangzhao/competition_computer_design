## 项目启停

```bash
scripts/start.sh   # 启动前后端（后台运行，日志写入 .run/）
scripts/stop.sh    # 停止前后端（读取 .run/*.pid 来 kill）
scripts/setup.sh   # 首次安装 Web 依赖
scripts/setup.sh --gpu  # 额外安装 COLMAP + 3DGS 重建环境
```

**不要用 `pkill` 或手动 kill 来停项目**，必须用 `scripts/stop.sh`，否则 pid 文件残留会导致下次启动跳过。

**注意**：`start.sh` 用 `bash -lc` 启动后端，会加载 `~/.bashrc` 中的 conda init，可能导致 conda Python 覆盖 `.venv` Python。脚本中已通过 `export PATH=.venv/bin:$PATH` 强制 venv 优先，不要删除这行。

## 手动操作 SQLite 注意事项

- **不要用 SQL 字符串拼接写入 JSON payload**。`knowledge_items` 等表的 `payload` 列存储 JSON 文本，如果 description 里包含 ASCII 双引号（`"`），手动 INSERT 会产出非法 JSON，导致后端 `json.loads()` 崩溃。
- 正确做法：通过 API 接口写入（会经过 Pydantic 校验 + `json.dumps()`），或者用 Python 脚本调用 `dump_payload()` 生成合法 JSON 后再插入。
- 如果必须手动 SQL 操作，用 `json_quote()` 或先用 Python 生成 JSON 字符串再粘贴。

**`.venv`（Web 服务 / 后端 FastAPI）**
- 标准 `venv` + `pip`，只装 `backend/requirements.txt` 里的包
- 启动/停止见上方脚本

**系统 Python `/root/miniconda3/bin/python`（3DGS 训练 / COLMAP）**
- AutoDL 镜像预装的 PyTorch conda 环境
- `reconstruction/setup_gpu.sh`（或 `scripts/setup.sh --gpu`）装进这个环境
- 两个环境不要混用
- `.env` 中 `ZHUYI_RECON_PYTHON` 必须指向这个路径

## 配置文件 `.env`

所有运行时配置都在项目根目录 `.env` 中，`scripts/start.sh` 会自动 `source` 它。**不要在脚本里 hardcode 覆盖环境变量**。

关键变量：
- `ZHUYI_DATA_DIR`：大文件存储根目录（留空则用 `backend/storage/`）
- `ZHUYI_RECONSTRUCTION_MODE`：`real`（真实重建）或 `mock`（假模型）
- `ZHUYI_RECON_PYTHON`：重建用的 Python 路径，必须是系统 conda Python（非 .venv）
- `ZHUYI_COLMAP_NO_GPU`：`0` 表示用 GPU COLMAP（本机已编译 CUDA 版）
- `DEEPSEEK_API_KEY`：DeepSeek 对话 API 密钥

## 存储架构

后端通过 FastAPI StaticFiles 直接 serve 大文件，**不使用软链接**。

- `ZHUYI_DATA_DIR` 环境变量指定存储根目录（如 `/root/autodl-tmp/zhuyi-data`）
- 有值 → `jobs/`、`generated/`、`models/` 在该目录下
- 无值 → 退回 `backend/storage/` 下的子目录
- `create_app()` 中 mount `/generated` 和 `/models` 静态路由
- `vite.config.ts` 中 proxy `/generated` 和 `/models` → `http://localhost:8000`
- `.splat` 文件已 gitignore，不提交到仓库

Splat 文件路径格式：
- 种子模型：`/models/xxx.splat`
- 重建/导入生成：`/generated/xxx.splat`

## Reconstruction Pipeline

The 3DGS reconstruction workflow has been fully practiced. **Read `reconstruction/重建流程实录.md` before touching any reconstruction code or commands** — it logs what was actually run, what worked, and what failed.

Key facts from experience:
- **Standard training: 10,000 iterations, `-r 4`** (quarter resolution). `-r 2` too slow for Web use case; 7k iterations too few, 30k takes hours.
- GPU COLMAP on headless server requires `xvfb-run -a` prefix
- COLMAP mapper may produce multiple submodels; `reconstruct.sh` auto-selects the largest by `images.bin` file size
- COLMAP uses `--ImageReader.single_camera 1`；`filter_images.py` 会自动将竖拍旋转为横拍统一分辨率
- COLMAP 注册图像数 < `MIN_IMAGES` 时脚本自动 fail，避免垃圾模型
- 3DGS coordinate system fix: use `--transform outdoor_arch` in `convert_ply_to_splat.py`
- After each reconstruction, run `python3 reconstruction/compute_camera_settings.py /root/autodl-tmp/<scene>` to get `cameraSettings` for `buildings.json` — do NOT manually guess camera position/lookAt
- Google Drive data is accessed via `rclone` with the pre-configured `gdrive:` remote (not gdown)
- Scene data lives in `/root/autodl-tmp/` (50 GB, persists across reboots)
- `.splat` files are gitignored; only `backend/data/buildings.json` metadata is committed

## Web Viewer — 3DGS 场景浏览

**控制方式：第一人称漫游（PointerLockControls）**，不是 OrbitControls。
理由：场景是真实空间（多座建筑、桥等），不是单个物体，不存在可以"围绕旋转"的中心点。

- `SplatViewer.tsx` 使用 `useBuiltInControls: false` 禁用库内置的 OrbitControls
- 改用 Three.js `PointerLockControls`：点击进入漫游，鼠标转视角，WASD 移动，ESC 退出
- 移动速度 `MOVE_SPEED = 0.02`（scene units/帧），场景尺度变化时需要调整

**cameraSettings（buildings.json 字段）**：
- `up`：**必须从 COLMAP 数据计算**，用 `compute_camera_settings.py`，不能手写 `[0,0,-1]`（不同场景的 COLMAP 世界坐标系 Y 轴不一定对齐重力）
- `position`：从 COLMAP 所有真实相机中选一个（选距离所有相机位置中点最近的那个），经 outdoor_arch 变换后的坐标。必须是真实存在的相机位置，不能是合成/插值的点
- `lookAt`：决定初始朝向，由 COLMAP 相机朝向计算，非 orbit pivot（已切换为第一人称，lookAt 仅影响初始视角）
- `.splat` 文件由我们的脚本生成，场景几何中心已平移到 `[0,0,0]`
- 外部 splat 文件（bonsai/church/st_pauls/taj_mahal）无 COLMAP 数据，cameraSettings 需用 SplatViewer 右下角 debug 按钮手动标定

**坐标系原理**（outdoor_arch）：
- 变换 `R_arch @ (pos - scene_center)` 应用于所有高斯球位置
- COLMAP 相机物理"上方"= `-R^T @ [0,1,0]`，经 `R_arch` 变换后得到 viewer 的 `cameraUp`
- 当 COLMAP 世界 Y 轴 ≈ 物理下方时，`cameraUp ≈ [0,0,-1]`，但这不是普遍规律

## 数据流：SQLite 为主，JSON 为种子

- 运行时所有数据读写走 SQLite（`backend/data/zhuyi.db`）
- `buildings.json` 等 JSON 文件**只在数据库为空时导入一次**作为种子数据
- 修改 `buildings.json` 不会影响已有数据库，改了也没用
- 要往运行中的系统添加模型，用 admin 接口 `POST /api/admin/import-model`（上传 .splat + 元数据，需 admin token）
- 或者删掉 `zhuyi.db` 让下次启动重新从 JSON 种子导入

## convert_ply_to_splat.py 裁剪选项

转换时可通过参数裁剪高斯体数量以减小文件体积：
- `--min-opacity 0.02`：移除低不透明度高斯体（推荐 0.02~0.1）
- `--max-scale 0.1`：移除过大的背景噪声高斯体
- `--max-gaussians 2000000`：按不透明度保留 top N

典型用法：`python3 convert_ply_to_splat.py input.ply output.splat --transform outdoor_arch --min-opacity 0.02 --max-scale 0.1`
